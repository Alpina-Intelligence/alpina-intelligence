AI and ML are generating an incredible amount of excitement. Current LLM technology is creating new applications of cheap intelligence, e.g. extract data from this scanned pdf and summarize it for me, and enabling more people to implement these applications. But these cool applications are built on the less-glamorous foundations of proper data engineering and management. That is why we are beginning our development and exploration of the Alpina Unified Intelligence Platform with our approach to data ingestion.

## Data Ingestion Challenges

One of the major challenges of data ingestion is the variety of data sources and formats. The data that we need to pull into our platform can live in different locations: the cloud, on-prem, file servers, different machines and networks... It can be in different formats: SQL and NoSQL databases, APIs, structured files like CSV and JSON, unstructured data like PDFs and images... It can be protected by different security measures: firewalls and IP whitelists, OAuth, API keys, integrated identity, role-based access control... The challenge of ingesting data is the challenge of designing and building a system that can handle this variety of sources, formats, and security measures in a way that is scalable, reliable, and maintainable.

This is a challenge common to every organization that is using data to achieve their goals (i.e. every organization). And it truly is a challenge! Ingesting and managing data for an organization takes expertise and resources, and this expertise and these resources are often outside the core competencies of most organizations. Given this, it is logical to conclude that there must be services that attempt to solve this problem (for a fee, of course). So this seems a logical first place to look for solutions to the challenge of data ingestion for micro, small, and medium enterprises (MSMEs).

## Managed ELT Platforms

For a small organization it may seem natural to reach for a managed service to solve this problem. After all, not even large organizations build their own e-mail services and rely on Outlook or Gmail. Doesn't similar logic apply here?

Unfortunately, the problem of data ingestion is more varied and complex. It is impossible for a managed service to cover all the combinations and permutations of sources, formats, interfaces, security measures, and organizational needs. There is still a lot of custom work and configuration that users need to do to actually get these services to work for them. These services provide the pre-built connectors and runtime environments for the data ingestion, but in my experience it is far from plug and play. In the era of AI, perhaps the value proposition of these services figuring out the mechanics of how to connect to the varied sources is less compelling than it used to be.

Fivetran and Hevo are two of the most popular managed ELT platforms. Databricks has a nascent Lakeflow Connect offering. These services all suffer from the same problems:
- pricing that scales unpredictably with volume and scale
- black box connector behavior that can be difficult to debug and troubleshoot
- imperfect coverage of sources, formats, and security measures
- lack of flexibility and customization

### Pricing

These services all use consumption-based pricing. We won't document the specifics here as the details change from time to time, but the general idea is that the more you use, the more you pay. With Databricks, you pay for the DBUs and the underlying compute resources. This can be economical for larger enterprises that are already using Databricks and can budget thousands of dollars per month for their data platform. But for the smaller organizations with small data volumes, becoming Databricks customers is a non-starter. Fivetran and Hevo have free tiers, which is always nice, and it isn't impossible that a micro or small enterprise could get by on the free tier. But as soon as you need to go beyond the free tier, the pricing can become prohibitive for smaller organizations. The pricing is also unpredictable, as it can vary based on the volume of data ingested and the number of connectors used. So it isn't impossible to get by using these services with a reasonable cost but this takes care, and now the incentive is to reduce the frequency of the updates, the number of sources, the amount of data being utilized by the organization. Our benchmark here is a cheap VPS ($10-$50) that can realistically handle quite a bit of I/O-bound data ingestion work. With this, the incentive flips: we are paying for a fixed amount of compute, so we want to keep it utilized and working.

### Black Box Connectors

While the pricing of these services could be manageable, having the connectors be black boxes that we are not able to inspect, debug, or troubleshoot can be a major problem. If a connector is not working in a way that we expect we are left communicating with the support team of the service. And for small orgs trying to get by on the free tier or at low cost, the support could be limited or non-existent. As someone used to stepping through code line by line with a debugger, inspecting state and logs as they execute this lack of visibility and control is a major frustration. 

### Imperfect Coverage

The number of pre-built connectors that these services provide is generally quite large and comprehensive. Lakeflow Connect has about 30, Hevo over 150, and Fivetran over 600. However, even using Fivetran I have run into issues connecting to certain sources. I was working on a project for a process manufacturing company that used Hexagon EAM for asset management and Infor LN for its ERP. These are large, important software systems with critical data to analyze for the organization. But we were left writing custom code to extract data from these systems and had to manage two systems for the data ingestion: Fivetran for sources with a pre-built connector and custom code for everything else.

### Lack of Flexibility and Customization

These services are designed to handle the most common sources and configurations, but in my experience they lack the flexibility and ability to customize that we inevitably run into when dealing with the complex and messy real world of data. In one example, we attempted to use Databricks Lakeflow Connect for MS SQL Server to pull in data from an on-prem SQL Server but we ran into issues due to the management of the data in the SQL Server. Many of the tables were 'snapshot' tables that were being truncated and replaced on a daily or monthly basis, blocking the use of CDC for the connector. Many of the tables also lacked primary keys, blocking the change tracking path. In another case, we wanted to use the Sharepoint connector from Fivetran to ingest files. The connector worked, but the files were Excel workbooks with complex layouts, not simple CSV or tabular data. So the connector couldn't just parse the files as tables. In both of these case the original source of the issue is how the data is being managed upstream of the ingestion, but we aren't always able to control that piece. I cannot force an organization to update their database management or use of Excel as a data store or how they layout their sheets. And while some of these changes would improve the data management of the organization, there is another sense in which the data ingestion tail should not be wagging the organizational dog. Folks in an organization are busy enough focusing on their core work without having the analytics team telling them to change their process to make data ingestion easier.

Fivetran does offer a 'Custom Connector' that allows users to build their own connectors using Python using their SDK and deploying them and running them in the Fivetran platform. This is nice in theory, but in practice the restrictions of this development setup can get in the way. Given the limitations of the pre-built connectors, we attempted to implement a custom connector to pull in Excel files from the file share of an Azure ADLS Gen2 Storage Account. We were able to implement the connector using the Fivetran SDK and even run it locally. But the connector failed when deployed on the Fivetran platform. The cause: missing dependencies. The connector is deployed with a `requirements.txt` file that specifies the dependencies but the Fivetran platform has a feature that does not install dependencies that are not imported somewhere in the connector code itself. I assume this is for efficiency or security reasons. However, the method for determining whether a dependency is imported must just be a simple text search which has the unintended consequence of not installing necessary dependencies that happen to have a different package name and import path. For example, our authentication used the `azure-identity` package, which is imported in the code as `from azure.identity import DefaultAzureCredential`. And the storage account client from the `azure-storage-file-share` package is imported as `from azure.storage.fileshare import ShareFileClient`. The Fivetran platform erroneously declined to install the dependencies and the connector runs fail due to missing dependencies. We were left firing off support tickets and going back and forth re-explaining the situation with little hope that Fivetran would deploy an update to their platform just for our little account. We abandoned this attempt to handle custom Excel file parsing within the Fivetran ecosystem.

These ELT platforms have a value proposition. They have done the heavy lifting of working out how to pull data from numerous sources, offer a GUI with configuration of pipelines just by clicking, and handle the infrastructure. However, they never cover 100% of cases and even just a few exceptions necessitates setting up parallel solutions. With modern LLMs and AI coding tools, the difficulty in writing connection logic has dropped significantly. And with these tools a GUI and click-based configuration is more of a hurdle than a help. And managed infra sounds great until you start pulling in millions of rows per month and are paying hundreds of dollars or more for compute that can reasonably be handled on a cheap box. This seems to point us towards writinng our own custom code to handle data ingestion.

## Custom Data Ingestion Code

On the other end of the spectrum we could go straight to writing our own custom code for everything. Of course we would still take advantage of existing libraries. We can use `httpx` for making HTTP requests to APIs, `pandas` for parsing and manipulating data, `sqlalchemy` for connecting to databases, `azure-identity` and `azure-storage-file-share` for connecting to Azure Storage Accounts, and so on. We control the code, we control infrastructure, and we control the costs. Sounds great, right?

Here is the catch: while we can use AI to generate code to connect to any API, we still need to understand how the API works to design this code correctly. Which endpoints exist, which objects support incremental queries, how custom filters behave, how deletes are handled, how pagination and cursors work, how authentication works... And we need to stay on top of changes to these APIs and to deal with the fact that the documentation is not always accurate. I've dealt with APIs where the documentation provides inaccurate information about available parameters. I am still not 100% confident in pointing an LLM at some documentation and expecting it to be able to generate correct code for handling the API. It can certainly help, but there is quite a bit of work involved in figuring this out. And it is the same for everyone; the amount of effort it takes to build the connection code has gone down, but this doesn't mean everyone has to solve the same problem when it has already been solved.

This also means that we need to build and maintain the system that orchestrates and runs our custom code. We need to trigger our pipelines, maintain their state, and we need to monitor and log their execution. Now we are writing boilerplate code to support our custom ingestion logic. Again, this may be a solved problem and not something we need to do from scratch. We need a middle ground between paying for an expensive managed service that is never going to cover every use case and spending all of our time writing code that has already been written and solving problems that have already been solved.
